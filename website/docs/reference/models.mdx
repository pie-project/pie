# Supported Models

What you can run depends on the [driver](./drivers) you choose.

- **`native` driver**: Pie's own engine. Each architecture needs an
  explicit implementation. The list below is the registry in
  `pie/src/pie_driver/model/__init__.py`.
- **`vllm` / `sglang` drivers**: Pie delegates inference. You get the
  full vLLM / SGLang model zoos. Some inferlet features (custom
  attention masks, page-trim, raw-logits sampling) are native-only.

## Architectures supported by the portable driver

The `portable` driver (`driver/portable/`) is a self-contained C++
binary built on raw `libggml` (vendored via llama.cpp). It supports
CPU, CUDA, Vulkan, Metal, and HIP backends selected at compile time.
Loads HuggingFace safetensors directly — no GGUF conversion.

fox jumps over the lazy dog. This sentence is a pangram…"). Base
checkpoints loop on the inferlet's chat-template wrap; use the
[`raw-completion`](#test-inferlets) inferlet for clean base-model
output.

## Test inferlets

`inferlets/text-completion` wraps the prompt with `system + user +
cue` chat template. Suitable for instruct-tuned models.

`inferlets/raw-completion` tokenizes the prompt verbatim with no
chat templating — required for clean output from base/pretrained
checkpoints (Llama-3.2-1B, Qwen2.5-3B-Base, etc.).

## Downloading

```bash
pie model download Qwen/Qwen3-0.6B                  # 0.6B default
pie model download Qwen/Qwen2.5-7B-Instruct
pie model download meta-llama/Llama-3.2-3B-Instruct
pie model download meta-llama/Llama-3.1-70B-Instruct
pie model download google/gemma-3-2b-it
```

```bash
pie model list           # show what's cached + which are pie-compatible
pie model remove <repo>  # delete from cache
```

## Configuring the model

```toml
[[model]]
name = "default"
hf_repo = "Qwen/Qwen3-0.6B"

[model.driver]
type = "native"
device = ["cuda:0"]

[model.driver.options]
gpu_mem_utilization = 0.8
```

Multi-GPU (tensor parallel × data parallel):

```toml
[model.driver]
type = "native"
device = ["cuda:0", "cuda:1", "cuda:2", "cuda:3"]
tensor_parallel_size = 4    # 4-way TP, 1× DP
```

If `device` has more entries than `tensor_parallel_size`, the leftovers form data-parallel replicas (e.g. `device = [...8 GPUs]` with `tensor_parallel_size = 4` runs as 2 DP groups of 4-way TP).

You can have multiple `[[model]]` blocks. The first entry is the implicit default; inferlets address the others via `Model::load("<name>")`. Each model's `device` list must be disjoint from the others.

## Quantization

The `native` driver supports float and quantized weight dtypes; the
choice maps to a torchao config under the hood
(`pie_driver/quantization.py`).

```toml
[model.driver]
activation_dtype = "bfloat16"

[model.driver.options]
weight_dtype = "float8"     # auto / float32 / float16 / bfloat16 / int4 / int8 / float8
```

| Value | Notes |
|---|---|
| `"auto"` | Inherit `activation_dtype`. |
| `"float16"` / `"bfloat16"` / `"float32"` | Plain floating-point weights. |
| `"int8"` | int8 weight-only quantization (torchao). |
| `"int4"` | int4 weight-only quantization (torchao). |
| `"float8"` | float8 weights (torchao). |

`activation_dtype` (one level up under `[model.driver]`) is independent, typically `bfloat16`.

vLLM and SGLang have their own quantization knobs; configure those in `[model.driver.options]` with `type = "vllm"` or `"sglang"` on the driver block.

## Adding model support

To support a new architecture under the native driver:

1. Add `pie/src/pie_driver/model/<name>.py` implementing the model.
2. Register it in `pie/src/pie_driver/model/__init__.py` with the matching
   HF `model_type`.

Or, for a one-off, point the model at the `vllm` or `sglang` driver. They
accept anything in the upstream backend's zoo.

If the architecture isn't in the upstream backends either, please
[open an issue](https://github.com/pie-project/pie/issues) with the
HuggingFace link.
